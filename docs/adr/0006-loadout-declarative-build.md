# ADR-0006 Loadout：声明式能力编组与分发（配置即产物，不碰权重）

| | |
| --- | --- |
| 状态 | Accepted |
| 日期 | 2026-07-02 |

## 背景

本地模型用户高度垂直化：政务用来管一类公文流程、网安用来打 CTF、另一些人用来做
家庭自动化或写作。每个人都有一个强烈的单一需求，而一个"什么都能干一点"的通用
agent 谁都服务不好。云端大模型可以靠体量硬扛全场景，本地模型不能——它的算力和
上下文都是稀缺的，必须为某个任务"配一套装备"才划算。

用户由此提出：能不能像 ComfyUI 或儿童拼图编程那样，把工具 / skills / MCP / 权限 /
机组全部可视化地热插拔组合，再"生成一份自定义的模型"，让不同人群拿到各自的专用版本。

这个洞察是对的，但原始形态里混了两件成本差几个数量级的事，且与设计铁律有正面碰撞，
必须先拆解、划边界，再定形态。

## 拆解：三个必须先厘清的边界

**1. "热插拔"的唯一可行形态是 authoring-time 编组、冻结、runtime 只加载。**
真正的运行时热插拔会在每次改动重建 prefix、打碎 KV 缓存、触发本地 30 秒级
re-prefill——直接违反 [原则 1 前缀稳定性](../architecture.md)。所以"热插拔"的正确含义
是"会话之间自由换装备（loadout）"，不是"在活着的对话里改 prefix"。换一套装备 = 开一个
新 session，这个代价可以接受。

**2. "自定义模型"现在 = 自定义 Build（配置产物），不碰训练管道。**
"模型"一词有三种成本天差地别的解释：(a) 配置 Build（persona + 工具分层 + 权限 + 机组，
零权重改动）；(b) Build 挂载睡眠周期产出的 LoRA（分期）；(c) 组合直接触发微调、
产出自定义权重（最重，触碰 provenance 红线，属 vision.md / M6）。本 ADR 只做 (a)，
并为 (b) 预留接口，(c) 明确排除。

**3. 这不是新架构，是 [ADR-0005 能力分层](0005-capability-disclosure.md) 之上的打包 + 编组层。**
一份 Loadout 就是声明"哪些能力进哪一层 + 权限档位 + 机组花名册 + 人设 + 记忆种子"。
底座（三层能力、append-only 披露纪律）已经在 ADR-0005 定死，本 ADR 只加一个
authoring 面和一个编译器。

## 决策：Loadout = 声明式格式 + 成本感知编译器 + 校验器 + CLI 向导

### Loadout 组合什么

一份 Loadout 是一个声明式清单（TOML），编组以下维度，全部是数据、无可执行代码：

| 维度 | 内容 | 约束 |
| --- | --- | --- |
| `metadata` | 名称、版本、作者、目标人群（ctf / gov / home ...）、描述 | 用于分发与检索 |
| `persona` | system prompt 片段 / 人设 | 进冻结前缀，计入 token 预算 |
| `tools` | 把内置工具按 core / cataloged 分层（见 ADR-0005） | core 层计入 <= 1400 token schema 预算 |
| `mcp` | 声明外部 MCP server 引用（foreign 层，完全惰性） | 只引用，不内联代码 |
| `permission` | 建议权限档位与按风险级的覆盖 | 见下方信任模型：导入时不自动继承 |
| `crew` | 机组花名册（lead / copilot / librarian / critic 各用哪个模型） | 是偏好，随硬件优雅降级 |
| `memory` | 记忆种子 / 领域知识 | 进可检索存储，不进前缀，不得撑爆预算 |
| `adapter` | 预留字段，未来挂载 LoRA（分期方案 b），现在恒为空 | 本 ADR 不实现训练 |

### 编译器职责（这是价值所在）

`kestrel loadout build` 不是简单读文件，它是编组纪律的强制点：

1. **解析引用**：把工具名对内置注册表校验、把 MCP 引用对已授权 server 校验，找不到即拒绝。
2. **算 token 账并拒绝超预算**：把 persona + core 层工具 schema 加总，超过 2.5k 总预算
   或 1400 schema 预算即编译失败并指出是哪个能力压垮的。这是"成本感知"的核心。
3. **冻结成确定性产物**：以稳定字节序输出，保证同一 Loadout 每次加载得到逐字节相同的
   前缀（否则违反前缀稳定性）。
4. **权限地板校验**：拒绝任何试图把破坏性 / 外联动作降到"用户不可拦截"以下的配置。

### 三条不可越界（把铁律焊死在这个特性上）

- **前缀稳定性**：编组只发生在 authoring-time，runtime 只加载冻结件。换 Loadout = 换
  session，永不在活对话里改前缀。
- **薄循环**：Loadout 只编组"能力 + 权限 + 人设 + 机组"，**绝不编排控制流**。它决定
  "带什么进战场"，不决定"怎么打"。任何走向 Dify / n8n / Flowise 式工作流引擎的诉求，
  在这里直接拒绝——那会让循环变厚，违反原则 3，就不再是 Kestrel。
- **安全（分发信任模型）**：一份共享的 Loadout 携带能力组合 + 人设，但**权限策略永不
  随导入继承**。导入他人 Loadout 时，本地默认回落到最严档，权限必须由本机用户重新确认。
  这样即便社区自由分享，也没有 Loadout 能夹带"自动放行 shell"这类后门——守住
  [AGENTS.md 第 7 节](../../AGENTS.md) 的红线。

### 形态：声明式格式 + 向导先行，可视化编辑器留作未来皮肤

先做 `.kestrel` 格式 + 编译器 + 校验器 + `kestrel loadout new` 交互向导。ComfyUI 式
可视化节点编辑器**现在不做**：底座（格式 + 编译器）稳了它才好长，且 WebUI 本已延后。
底座对了，皮肤是后话。

## 差异化：别人给你看连线，我们给你看代价

编组界面（先是 CLI 向导，未来是皮肤）实时显示 token 预算表与前缀影响：加一个能力时
告诉你它进 core 层会占多少预算、进 cataloged 层只追加到尾部不污染前缀。把 Kestrel
最硬的约束（2.5k 预算、前缀稳定）直接变成 UX——这是一台**成本感知的能力编译器**，
不是又一个连线玩具。没有别的 agent 会做，因为没有别的 agent 在乎这个预算。

## 被否决方案的最强论点（诚实记录）

- **现在就做可视化节点编辑器**：最眼前一亮、传播力最强，是"火"的直接抓手。否决理由：
  前端工作量巨大、节点图天然诱导滑向工作流引擎、WebUI 已延后。保留为未来皮肤，套在
  声明式底座之上，不是放弃。
- **组合直接触发训练、产出自定义权重**：这是最强护城河——配方可分享、权重是你独有的。
  否决理由：最重最慢，且触碰"训练只能由用户亲手输入触发"的 provenance 红线。归入
  [vision.md](../vision.md) 的睡眠周期与 M6，通过 `adapter` 预留字段分期接入，而非现在硬上。

## M1 与落地边界

- **M1 不动**：M1 只有 4 个 core 工具全部常驻前缀，无需 Loadout。
- Loadout 编译器依赖能力分层，而分层按 ADR-0005 在 M4 才展开，故 Loadout 与 M4 同期或
  其后落地；在此之前先固化本决策与格式草案，预留扩展点。
- `metadata.adapter` 字段现在即在格式里留空占位，为分期方案 b（挂载 LoRA）保留接口，
  避免将来破坏格式兼容。

## 重开条件

- 社区实际分享 Loadout 后发现声明式不够表达某类编组 -> 评估是否需要更强的组合原语，
  但**绝不引入控制流**（那是另一类产品）。
- 权重定制的需求强烈到值得提前 -> 走新 ADR，把 vision.md 的睡眠周期产物正式接入
  `Loadout.adapter`，而不是在本 ADR 里偷偷扩张。
- CLI 向导体验不足、用户强烈要可视化 -> 评估节点编辑器作为皮肤落地，底座格式不变。
