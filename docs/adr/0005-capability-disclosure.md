# ADR-0005 能力披露分层：渐进式披露，但只许向尾部追加

| | |
| --- | --- |
| 状态 | Accepted |
| 日期 | 2026-07-02 |

## 背景

固定 token 开销必须压到 2.5k 以内（原则 2）。M1 只有 4 个工具、在预算内。
但随着 skills、更多工具、MCP 接入，工具 schema 会膨胀成我们批评 Claude Code
的那个问题（14.3k 固定开销）。用户提出用渐进式披露（progressive disclosure）
按需暴露能力，问是否有更好方法。

## 关键洞察：本地场景下，渐进式披露有一个隐藏陷阱

渐进式披露在云端很成熟（Anthropic 自家 harness 的 ToolSearch 即是）。但**朴素做法
会与原则 1（前缀稳定性）正面冲突**：

- KV 前缀缓存的有效范围是"从头到第一个变化的字节"。
- 若在对话中途把新工具 schema 注入到 **system prompt 或已缓存的前缀里**，
  前缀改变 -> KV 缓存全灭 -> 全量 re-prefill（本地 30 秒级）。
- 云端按 token 计费，动态注入无所谓；本地动态前缀是灾难。

所以问题不是"要不要渐进式披露"，而是"如何在不破前缀的前提下披露"。

## 决策：三层能力 + append-only 披露纪律

以 KV 缓存边界划层：

| 层 | 内容 | 位置 | 缓存行为 |
| --- | --- | --- | --- |
| **常驻（Core）** | read / edit / shell / search 等每轮几乎必用的少量工具 | 静态前缀 | 永久缓存 |
| **在册（Cataloged）** | skills、专用工具：清单里只放"名称 + 一行描述" | 前缀放清单，正文按需披露 | 披露时追加到**消息尾部** |
| **外部（Foreign）** | MCP：只列服务器名，工具枚举按需 | 完全惰性，绝不进前缀 | 同上，尾部追加 |

**铁律（本 ADR 的核心）：披露只允许向消息历史的尾部 append，永不改写已缓存的
前缀。** 于是首次披露某能力只付一次"尾部 re-prefill"（小），此后该能力随
append-only 历史被纳入新前缀、继续缓存。你永远不为没用到的能力付前缀税，
也永远不因披露而打碎核心缓存。

### 比朴素渐进式披露更好的一招：书记预取（Librarian pre-fetch）

朴素披露要模型花一轮调用 `search_tools` 去发现能力（多一次往返 = 多一次本地延迟）。
我们有 Qwen3-Embedding-4B（机组的书记）：在轮次开始时用嵌入检索，把与用户请求
最相关的"在册能力"**主动预取**、一次性追加到尾部，模型通常无需专门的发现往返。
找不到时再回退到显式 `use_capability`。这把"渐进式披露"从被动（模型自己去搜）
升级为主动（书记提前铺好），且同样遵守 append-only 纪律。

## M1 的落地边界

M1 只有 4 个 core 工具，全部常驻前缀——这是最简且正确的形态，本 ADR 不改 M1 代码。
分层在 M4（工具/skills/MCP 增多）才需要。现在固化决策与纪律，并预留扩展点：

- `ToolSet` 已支持"注册但不一定全部进前缀"的结构；
- `CompletionRequest` 未来区分"前缀工具"与"尾部披露工具"；
- 披露产物一律走消息尾部，与异地压缩、影子槽预热共享同一条"不碰前缀"纪律。

## 重开条件

- 实测发现书记预取召回率过低、模型频繁需要显式发现往返 -> 调整预取策略或扩大常驻层。
- 某能力虽属"在册"却几乎每轮都用 -> 提升为常驻（放回前缀），用数据说话。
